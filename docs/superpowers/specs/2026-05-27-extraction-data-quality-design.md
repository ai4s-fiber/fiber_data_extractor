# Extraction Data Quality Design

## Goal

Improve the V7 fiber literature extraction pipeline so strong models produce auditable, reusable material data instead of wider but sparse numeric tables.

The final candidate record schema remains unchanged. The pipeline will add internal layers that make final rows inherit paper, sample, composition, process, structure, evidence, and cleaned value information in a predictable way.

## Scope

- Add a `sample_card` normalization layer derived from the existing sample catalog and composition/process facts.
- Add a `result_fact` normalization layer derived from performance facts.
- Generate final records by merging one sample card with one result fact.
- Classify metrics into `Core`, `Secondary`, or `Narrative` priority.
- Clean values into raw/operator/clean/unit metadata while preserving final 40-column compatibility.
- Enforce evidence and source-location quality rules.
- Export review workbooks with separated core records, secondary records, sample cards, result facts, and quality report sheets.
- Update the front-end review page so humans review a compact core queue first, with full record details in a drawer.

## Non-Goals

- Do not change the final 40/41 exported field names.
- Do not add paper-specific hardcoded rules for the current test article.
- Do not treat Excel as the primary human review surface.

## Data Flow

1. Stage 1 creates paper metadata and a sample catalog.
2. Stage 2 extracts facts from text, tables, and figure captions.
3. Stage 3 assigns facts to sample IDs.
4. New normalization converts samples into sample cards and performance facts into result facts.
5. Final candidate records are generated by field-name merge from sample card plus result fact.
6. Quality report calculates counts, missing rates, evidence gaps, rough source locations, schema alignment, and core metric coverage.

## Export Workbook

- `Core_Final_Records`: final schema, only core metrics by default.
- `Secondary_Final_Records`: final schema, secondary characterization metrics.
- `Sample_Cards`: one row per sample card with composition/process/structure fields.
- `Result_Facts_QA`: one row per normalized result fact with assignment, priority, cleaned value, and export target.
- `Quality_Report`: extraction quality diagnostics for the run.

## Front-End Review

The review page defaults to a compact core-review table:

- visible columns: sample, metric, value, unit, source, evidence, confidence, review status.
- details drawer: paper metadata, sample card, cleaned result fact, evidence, reviewer fields, and full candidate record.
- filters: core/secondary, status, issue type, sample, metric.

This keeps the human review workflow short while preserving full traceability.

## Validation

- Final records are schema-built by field names, never by list position.
- Missing fields are filled as empty strings.
- Records with missing evidence or rough source locations become `uncertain`.
- Value cleanup removes units from `performance_value` where possible and stores raw cleanup metadata in internal QA rows and reviewer comments.
- Metric priority is dictionary-based and generic across papers.
